protocol

OPAS

A skill is a YAML file describing pedagogy, persona, learning phases, and corpus references. The runtime loads any conformant skill and runs it as a tutoring agent — Socratic dialogue, grounded in your corpus, structured by phases. Model-agnostic: Claude, GPT-4, Gemini all work.

The Soul

Two files. A YAML constitution defining five always-on rules + 16 distress patterns + 9 harm patterns. And a Python enforcer that scans every learner message and every agent reply against those patterns deterministically. Drop into any agent in 5 lines.

How they fit together

Every learner message routes through the Soul before reaching the model; every agent reply routes through the Soul before reaching the learner. OPAS provides the tutoring structure; the Soul provides the safety floor. Each primitive works without the other — they compose when you use both.

Why two protocols, one repo

We built these together while shipping a tutoring product (Poppy) and noticed they're each useful in isolation.

OPAS solves "how do I describe a tutoring skill in a way that survives the next model release?" — by making the skill data, not code, and the LLM swappable.

The Soul solves "how do I make a student-facing agent that doesn't fail catastrophically when the student is in crisis?" — by adding a deterministic regex layer below the model.

You can adopt OPAS without the Soul (e.g., adult corporate L&D where wellbeing intercepts aren't relevant), or the Soul without OPAS (e.g., you're already running your own tutoring stack and just want the safety layer). Both work standalone.

Three commands to run it

Full setup walkthrough lives at /quickstart — covers Supabase, env vars, and minting a learner token. The fast version:

# 1. Clone
git clone https://github.com/kriziadelgado88/OPAS.git
cd OPAS/agent && pip install -r requirements.txt

# 2. Configure
cp .env.example .env  # fill in Supabase + Anthropic keys

# 3. Run
uvicorn app.main:app --reload --port 8001
